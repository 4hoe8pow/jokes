## 1. DNSの基本概念

### ゾーン (Zone)
- **定義**: ドメインに対応する名前空間の管理単位。
- **内容**: ゾーン内のリソースレコード（A, MX, NSなど）を保持。
- **管理責任**: SOA (Start of Authority) レコードによって責任者やシリアル番号、更新ポリシーが明示される。
- **役割**: DNSの分散管理を可能にする「境界」の役割を担う。

### 権威 (Authoritative)
- **定義**: ゾーン情報のオリジナルを保持し、その正本として回答するサーバ。
- **特徴**:
  - 正式な回答を返す（権威応答 / AAフラグあり）。
  - キャッシュサーバが返す非権威応答と区別される。
- **重要性**: 権威サーバが侵害されると、そのドメイン全体の正当性が失われる。

## 2. 主なDNSレコードとセキュリティ上の効果

| レコード種別 | 用途 | セキュリティ上の効果・リスク |
|--------------|------|------------------------------|
| **A / AAAA** | ホスト名 → IPv4/IPv6 | キャッシュポイズニングの標的。正規サイト偽装に直結。 |
| **NS** | ゾーンの権威サーバ指定 | 不正変更されるとドメイン全体の解決が攻撃者支配に。 |
| **SOA** | ゾーン管理情報 (シリアル番号等) | 改ざんによりレプリカ更新妨害や情報漏洩の恐れ。 |
| **MX** | メール配送サーバ指定 | 乗っ取りでメール盗聴・拒否・不達攻撃。 |
| **CNAME** | 別名定義 | サブドメインテイクオーバーの温床。 |
| **PTR** | 逆引き (IP → ホスト名) | 不整合でスパム判定リスク。信頼性に影響。 |
| **TXT (SPF)** | メール送信許可IP宣言 | MAIL FROM のなりすまし防止。 |
| **TXT (DKIM)** | メール署名の公開鍵 | 改ざん検知。整合性保証。 |
| **TXT (DMARC)** | SPF/DKIMの整合性ポリシー | From表示アドレスのスプーフィング対策。 |
| **CAA** | 証明書発行許可CA指定 | 誤発行・不正発行防止。PKI連携強化。 |
| **DNSKEY / RRSIG** | DNSSEC用の鍵と署名 | 応答改ざんを暗号学的に検知可能。 |

## 3. 再帰問い合わせとキャッシュポイズニング

### 再帰問い合わせの流れ
1. クライアント → リゾルバへ名前解決要求。
2. リゾルバが root → TLD → 権威サーバ へ段階的に問い合わせ。
3. 応答をキャッシュしてクライアントへ返答。
4. 次回以降はキャッシュから応答（TTL有効期限内）。

### キャッシュポイズニング
- 攻撃者がリゾルバの問い合わせに対して **偽の応答を権威より先に注入**。
- 標的: 主に **Aレコード**（正規IPを偽IPにすり替える）。
- 成果: ユーザを攻撃サイトへリダイレクトし、フィッシングやマルウェア配布が可能になる。

## 4. カミンスキー攻撃と従来型キャッシュポイズニング

| 項目 | 従来型キャッシュポイズニング | カミンスキー攻撃 |
|------|----------------------------|------------------|
| 手法 | 偶発的にトランザクションIDやポート番号を推測 | ランダムサブドメインを大量生成し、応答競合を増幅 |
| 成功率 | 低い（推測成功に依存） | 高い（大量試行により高速成功可能） |
| 攻撃対象 | 単一問い合わせ | サブドメイン全体 |
| 影響範囲 | 局所的 | ドメイン全体のキャッシュを汚染可能 |
| 対策 | - トランザクションIDランダム化<br>- ソースポートランダム化 | 上記に加え **DNSSEC導入**、サブドメインクエリへの制御 |

## 5. UDPとTCPの違い（偽装容易性）

| 項目 | UDP | TCP |
|------|-----|-----|
| 通信方式 | コネクションレス | コネクション型 (3-way handshake) |
| 検証要素 | 16bitトランザクションID + 送信元IP/ポート | シーケンス番号同期、接続確立 |
| 偽装難易度 | 容易（IDとポート推測で成立） | 難しい（正しいシーケンス番号を事前把握困難） |
| 典型用途 | DNS, VoIP, ストリーミング | Web, Mail, ファイル転送 |

## 6. DNSSECによる改ざん防止

- **署名付き応答**: 各レコードセットに対しRRSIGが付与される。
- **検証プロセス**: DNSKEYにより署名を検証し、改ざんや偽応答を検出。
- **信頼の連鎖**: Root → TLD → ドメイン へ公開鍵を鎖のように委譲。
- **効果**:
  - 攻撃者が偽応答を注入しても署名が不一致 → 検出され無効化。
  - キャッシュポイズニングやカミンスキー攻撃に対する強固な防御策。

## 7. メール関連セキュリティ（SPF）

### SPFの仕組み
- DNSに **「このドメインからメールを送信できるIP範囲」** を宣言。
- 受信MTAが送信元IPと照合し、不正IPを拒否または隔離。

### SPFレコード例

```txt
example.com. IN TXT "v=spf1 ip4:203.0.113.0/24 include:_spf.service.net -all
```
- `v=spf1`: SPFバージョン識別
- `ip4:203.0.113.0/24`: 許可IP範囲
- `include:`: 外部サービスを許可
- `-all`: 上記以外は拒否

### SPFの効果と限界
- ✅ **MAIL FROM のなりすまし防止**
- ❌ **ヘッダFrom偽装は防げない** → DMARCと併用必須
- ❌ **転送メールでSPF失敗する問題** → SRSやDKIMで補完

## 8. 代表的対策まとめ

| 攻撃 | 対策 |
|------|------|
| キャッシュポイズニング | ソースポートランダム化・トランザクションIDランダム化・DNSSEC導入 |
| カミンスキー攻撃 | 同上 + サブドメイン応答の厳格化、DNSSEC |
| メールスプーフィング | SPF, DKIM, DMARC の三点セット |
| 証明書誤発行 | CAAレコードによる発行CA制限 |
| サブドメインテイクオーバー | CNAME利用監査、不要レコード削除、外部サービス管理徹底 |
